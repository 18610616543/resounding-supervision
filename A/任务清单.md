## 任务清单

### 任务 A-1：GitHub 仓库初始化

- **目标**：创建公开仓库，搭建标准化项目结构
- **预计时间**：30分钟
- **具体步骤**：
  1. 在 GitHub 创建公开仓库 `EchoCare`
  2. 设置仓库描述：`情感陪伴机器人管理控制台 - 开源软件开发课程实践`
  3. 添加 `.gitignore`（Node/Web 模板）
  4. 设置 Branch Protection Rules（main 分支：要求 PR + Review 才能合并）
  5. 提交现有4个代码文件：`index.html`、`css/style.css`、`js/app.js`、`echo-care.html`
  6. 邀请4名团队成员为 Collaborator
- **验收标准**：仓库公开可访问，4人均有 push 权限，main 分支受保护

---

### 任务 A-2：README.md 撰写

- **目标**：完整介绍项目，让任何人能理解和运行
- **预计时间**：40分钟
- **必须包含内容**：
  1. 项目名称 + 一句话简介
  2. 核心功能列表（管理控制台 + 情感聊天两个模块，各列出3-5个功能点）
  3. 项目截图（至少1张：控制台界面）
  4. 技术栈说明（HTML/CSS/JS，Web Speech API，localStorage）
  5. 快速开始（3步：clone → 打开 index.html → 使用）
  6. 项目结构说明
  7. 团队成员列表
- **验收标准**：markdown 格式正确，GitHub 渲染正常，所有链接可点击

---

### 任务 A-3：LICENSE 选择与添加

- **目标**：选择合适的开源许可证并说明理由
- **预计时间**：15分钟
- **具体步骤**：
  1. 选择 MIT License（宽松、简短、被广泛接受）
  2. 在 GitHub 创建仓库时勾选或手动添加 `LICENSE` 文件
  3. 在 README 中注明许可证类型
- **验收标准**：LICENSE 文件存在于仓库根目录

---

### 任务 A-4：CONTRIBUTING.md 撰写（贡献指南）

- **目标**：明确协作者如何参与项目
- **预计时间**：20分钟
- **必须包含内容**：
  1. 如何提交 Issue（Bug报告 / 功能请求）
  2. 分支策略（feature 分支 → PR → main）
  3. 提交信息规范
  4. PR 模板说明
  5. Code Review 流程（至少1人 Review → 通过后合并）
- **验收标准**：内容清晰，团队4人均能读懂并执行

---

### 任务 A-5：Issue 创建与分配

- **目标**：将全部任务拆分为5个 GitHub Issue 并分配
- **预计时间**：20分钟
- **具体步骤**：
  1. 创建 Issue #1：`[A] 仓库初始化 + 项目文档 + 版本发布` → assign 自己
  2. 创建 Issue #2：`[B] Agent数据校验增强 + Bug修复` → assign 成员B
  3. 创建 Issue #3：`[C] 预设人格模板扩充 + 润色引擎优化` → assign 成员C
  4. 创建 Issue #4：`[D] 对话数据库扩展 + 情感匹配优化` → assign 成员D
  5. 创建 Issue #5：`[E] 分享码健壮性 + 全流程测试` → assign 成员E
  6. 每个 Issue 标题清晰、有简要描述、有明确的验收条件
- **验收标准**：5个 Issue 在 GitHub Issues 列表可见，assignee 正确

---

### 任务 A-6：Review 与协调（半天3）

- **目标**：审核成员 PR，推动合并
- **预计时间**：60分钟
- **具体步骤**：
  1. 至少 Review 成员C和成员D的 PR
  2. 每条 Review 评论具体指出代码问题或改进建议（不只是"LGTM"）
  3. 在半天3开始前检查所有成员 PR 是否已提交
  4. 对延迟的成员点对点提醒
- **验收标准**：至少2条 Review 评论有实质内容；全部 PR 在半天3结束前合并

---

### 任务 A-7：Release 发布（半天3收尾）

- **目标**：创建正式版本发布
- **预计时间**：20分钟
- **具体步骤**：
  1. 所有 PR 合并后，在 main 分支打 Tag：`v1.0.0`
  2. 创建 GitHub Release，关联 Tag
  3. 编写 Release Notes（列出本次版本的改进点，参考各成员 PR 描述）
  4. 更新 README.md 中项目状态为 Released
- **验收标准**：Release 页面可见，含 Tag、Release Notes、下载链接
