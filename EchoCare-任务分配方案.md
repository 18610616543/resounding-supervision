# EchoCare 情感陪伴机器人 — 开源项目任务分配方案

## 时间约束：三个半天（约12小时/人）

---

## 一、项目概述

**项目名称**：EchoCare · 情感陪伴机器人  
**核心功能**：智能体管理控制台 + 情感陪伴聊天交互

**已有代码资产（价值：省去从零开发）**：
| 文件 | 行数 | 功能覆盖 |
|------|------|----------|
| `index.html` | 347 | 控制台页面结构 |
| `css/style.css` | 435 | 暗色主题完整样式 |
| `js/app.js` | 1216 | Agent CRUD、认证、人格双分支、AI润色、预览引擎、分享码 |
| `echo-care.html` | 1174 | 情感分析、对话系统、TTS、语音识别、角色动画、主动陪伴 |

---

## 二、三半天节奏总览

| 时段 | 目标 | 关键动作 |
|------|------|----------|
| **半天1** | 仓库就绪 + 任务拆分 + 各自认领 | 组长建仓库、创建5个Issue；全员 fork/clone、搭环境、跑通现有代码 |
| **半天2** | 各自开发 + 提交PR | 每人围绕自己认领的 Issue 完成代码修改、自测、提PR |
| **半天3** | 交叉Review + 修复 + 集成发布 | 互相Review、修问题、合并、打Tag发Release |

---

## 三、五人具体分工

核心思路：不写新代码，而是在现有代码基础上每人做**可追溯的、有意义的改进**，并通过 GitHub Issue/PR/Review 流程落地。

---

### 成员A（组长）：项目架构 + IN/OUT 文档 + 版本发布 + Review 协调

| 半天 | 任务 |
|------|------|
| **半天1** | ① 创建 GitHub 公开仓库，邀请4人加入；② 提交初始代码（4个文件）；③ 编写 `README.md`（含功能介绍、截图、运行步骤）；④ 选择并添加 `LICENSE`（MIT）；⑤ 编写 `CONTRIBUTING.md`（PR规范、分支策略）；⑥ 创建5个 Issue 并分配给对应成员 |
| **半天2** | ① 编写 `changelog.md` 或 Release Notes 草稿；② 检查各成员 PR 进度，协调阻塞问题；③ 检查 README 是否与最终代码一致；④ 辅助任一遇到困难的成员 |
| **半天3** | ① 作为 Reviewer 审核至少2个成员的 PR；② 汇总修复意见；③ 全部合并后打 Tag `v1.0.0`；④ 创建 GitHub Release 并发布 Release Notes；⑤ 更新 `README.md` 中项目状态为 Released |

**预期产出**：
- GitHub 公开仓库（含 README、LICENSE、CONTRIBUTING）
- 5个 Issue 记录
- 至少2条 Review 评论记录
- Release v1.0.0（Tag + Release Notes）

**有效GitHub证据**：仓库URL、README、LICENSE、Issue列表、Review评论、Release记录

## 四、半天1 行动清单（所有人，2小时内完成）

| 步骤 | 谁 | 做什么 | 产出 |

| 4 | 全员 | 在对应 Issue 下回复"认领，预计半天2完成" | Issue 评论 |
| 5 | 全员 | 创建自己的 feature 分支：`feature/xxx` | feature分支 |

## 五、Git 协作规范（简易版）

| 事项 | 规范 |
|------|------|
| 分支命名 | `feature/模块名`，如 `feature/agent-crud` |
| 提交信息 | 中文即可，格式：`做了什么` |
| PR 标题 | `[对应Issue编号] 简要描述` |
| PR 合并 | 至少1人 Review 通过 → 由 PR 作者自行 merge |
| 禁止事项 | 不直接 push 到 main；不 force push |

---

## 六、你的角色选择建议

**你（组长）最适合做成员A**，原因：

1. 仓库、Issue、Release 都由你发起和收尾，天然掌控全局节奏
2. 文档工作量适中，三个半天刚好：半天1建仓库+README，半天2写CHANGELOG+协调，半天3 Review+发布
3. Review 协调权在你手里，可以推动所有人按时完成
4. 课程报告中大量内容（项目目标、版本范围、协作方式等）与你直接重合，写报告不用到处要材料
5. 其他4人的代码改进都在现有代码基础上做**增量修改**（不是从零开发），在半天2内完全可行

**其余4人按B→C→D→E分配即可，改动都是对现有代码的增强，每个半天2最多2小时编码量。**
